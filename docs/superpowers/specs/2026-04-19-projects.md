# Projects — Design Spec

**Date:** 2026-04-19
**Author:** Andrew + brief brainstorm
**Status:** Ready for plan

## Goals

Introduce `Project` as a grouping layer between `Organization` and `Agent`. Hierarchy becomes `Org → Project → Agent → (Env at trajectory-level) → (Version at AgentVersion-level)`.

## Non-goals

- Multi-org support (spec: one org per deployment).
- Per-project ACL / role separation. Projects are an organizational primitive for v1, not a permission boundary.
- Cross-project move-at-ingest-time: changing an agent's project is a UI action, not an SDK signal.
- Version-level project reassignment. AgentVersions follow their Agent.
- Audit trail of mutations (name changes etc.). Noted as a follow-up — UUIDs stay stable so a separate audit table can be bolted on later without schema churn.

## Data model

**New table: `projects`**
- `id UUID PK` — immutable
- `org_id UUID FK → organizations(id)` — tenancy
- `name VARCHAR(255) NOT NULL` — display name, mutable
- `slug VARCHAR(255) NOT NULL` — unique per org, used in URLs + API
- `description TEXT NULL`
- `color VARCHAR(32) NULL` — Aether Dusk token name (e.g. "aether-teal"). Optional; UI falls back to neutral.
- `created_by_user_id UUID FK → users(id) ON DELETE SET NULL`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`

Unique constraint: `(org_id, slug)`.

**Agent change:**
- Add `agents.project_id UUID NOT NULL FK → projects(id) ON DELETE RESTRICT`.
  - `RESTRICT` — you must reassign agents before deleting a project.

**Migration 0016:**
1. Create `projects` table.
2. For every existing org, insert a `(name="Default", slug="default")` project owned by the org's first admin (or NULL `created_by_user_id` if no users yet).
3. Add `agents.project_id` nullable.
4. Backfill `agents.project_id` = that org's default project.
5. Set `agents.project_id NOT NULL`.

## API

All routes are `require_user()`-gated and `user.org_id`-scoped.

- `GET /api/projects` → list projects with agent counts.
- `POST /api/projects` → `{name, slug?, description?, color?}` — slug auto-generated from name if missing; 409 on duplicate.
- `GET /api/projects/{slug}` → detail with agent list.
- `PATCH /api/projects/{slug}` → `{name?, description?, color?, rename_to_slug?}` — slug mutation supported.
- `DELETE /api/projects/{slug}` → 409 if any agents are assigned. Can't delete the `default` project.

**Agent endpoints:**
- `GET /api/agents` gains optional `?project=<slug>` filter.
- `POST /api/agents` payload accepts `project_slug?` — defaults to `"default"`.
- `PATCH /api/agents/{name}` accepts `project_slug?` for reassignment.
- Agent list + detail responses include:
  ```json
  "project": { "id": "...", "slug": "default", "name": "Default", "color": null }
  ```

## UI

**`/agents` page:**
- Above the agents table: a project picker chip row (`All projects · Default · Prod-bots · …`). Selected state highlights; click toggles filter. Persists in URL `?project=<slug>`.
- Table gains a `Project` column (small color-dot + slug).
- "+ Add agent" modal gets a `Project` select (defaults to current filter, or `default` if filter is "all"). Inline "+ new project…" shortcut opens a mini-form.

**Settings page:**
- New section `/settings/projects` — table of projects with name, slug, agent count, row actions (edit / delete). "+ New project" button. Modal form for create/edit (name, slug, description, color via chip swatches).

**Nav config:**
- Add `{ id: "projects", label: "Projects", href: "/settings/projects", group: "workspace" }` to `SETTINGS_SECTIONS`.

**No changes to:**
- Queue / Clusters / Dashboard / History. Projects are a v1 organizational tool, not a filter on other views yet.

## SDK

Python SDK `langperf.init()` signature grows:

```python
def init(
    *,
    api_token: Optional[str] = None,   # required
    agent_name: Optional[str] = None,  # NEW — preferred over service_name
    service_name: Optional[str] = None, # deprecated alias for agent_name (v1 compat)
    environment: Optional[str] = None, # unchanged
    version: Optional[str] = None,     # NEW — maps to service.version resource attr
    endpoint: Optional[str] = None,
) -> TracerProvider: ...
```

- If both `agent_name` and `service_name` are passed → `agent_name` wins, log a deprecation warning.
- `version` attaches as `service.version` on the OTel Resource → flows through to `AgentVersion.package_version` via existing resolver logic. No new server code needed.
- `environment` stays mapped to `deployment.environment` (already in place).
- `agent_name` is advisory — the token-bound `agent_id` on the receiver is authoritative. If they disagree, we ingest under the token's agent and ignore the name.

## Breaking changes

- **DB**: adds `project_id` to agents as NOT NULL. Migration seeds + backfills, so existing installs upgrade transparently.
- **API**: additive. `/api/agents` responses gain a `project` field (non-breaking for clients that ignore extra keys).
- **SDK**: additive kwargs; `service_name` kept with deprecation warning.

## Testing

- API: project CRUD tests (create, list, conflict on dup slug, filter, reassign agent, delete-with-agents rejected, delete-default rejected).
- Ingest: token-based ingest with a `project_id` agent lands trajectory under that project's agent — verified by checking the agent row's project.
- Playwright: `/agents?project=default` filters the table; modal "+ new project" flow; settings page CRUD.
